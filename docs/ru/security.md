# Безопасность

## Модель аутентификации

### Management API (Admin Token)

Management API защищён статическим bearer-токеном:

```
Authorization: Bearer <ADMIN_API_TOKEN>
```

- Настраивается через переменную окружения `ADMIN_API_TOKEN`
- Используется для server-to-server коммуникации
- Constant-time сравнение токенов предотвращает timing-атаки
- Токен считывается один раз при старте
- Если `ADMIN_API_TOKEN` не задан, Management API не защищён; используйте это только для локальной разработки

!!! warning "Храните admin-токен в секрете"
    Admin-токен даёт полный доступ к созданию/удалению диалогов и управлению участниками. Никогда не раскрывайте его фронтенд-клиентам.

### Chat API (JWT-аутентификация)

Chat API поддерживает опциональную JWT-аутентификацию. При включении (`JWT_AUTH_ENABLED=true`) MTChat валидирует токены на сервере:

**Как работает:**

- REST API: токен передаётся в заголовке `Authorization: Bearer <token>`
- WebSocket: токен передаётся как query-параметр `?token=<token>`
- ID пользователя извлекается из JWT claim, указанного в `JWT_USER_ID_CLAIM` (по умолчанию `sub`)
- Подпись валидируется по алгоритму HS256 с использованием `JWT_SECRET`
- Срок действия (`exp`) НЕ проверяется -- время жизни токена управляется вашим приложением

**Формат токена:**

```json
{
  "sub": "user-id-here",
  "iat": 1234567890
}
```

**Конфигурация SDK:**

```typescript
const config: MTChatConfig = {
  baseUrl: 'https://chat.example.com',
  userId: user.id,
  token: jwtToken,  // JWT из вашей системы аутентификации
  // ...
}
```

!!! tip "Рекомендация для продакшена"
    Включите JWT-аутентификацию в продакшене (`JWT_AUTH_ENABLED=true`), чтобы только аутентифицированные пользователи могли получить доступ к чату. Установите сильный `JWT_SECRET` (минимум 32 символа).

**Legacy-режим (JWT выключен):**

Когда `JWT_AUTH_ENABLED=false` (по умолчанию), Chat API идентифицирует пользователей через query-параметр `user_id`. В этом режиме ваше приложение отвечает за то, чтобы только аутентифицированные пользователи имели доступ к SDK с корректным `user_id`.

## Контроль доступа

### На основе участников

- Только **прямые участники** могут читать/отправлять сообщения
- Неучастники получают `403 Forbidden`
- Потенциальные участники видят метаданные диалога, но не сообщения

### На основе scope

- Список "Доступные" фильтруется по [scope-правилам](guide/scope-matching.md)
- Пользователи видят только диалоги, соответствующие их `scope_level0`, `scope_level1` и `scope_level2`

## Валидация ввода

Все текстовые поля проверяются на длину:

| Поле | Макс. длина | Описание |
|------|-------------|----------|
| Содержимое сообщения | 50 000 символов | HTML-тело сообщения |
| Заголовок диалога | 500 символов | Название диалога |
| Отображаемое имя | 200 символов | Имя участника |
| Компания | 200 символов | Название компании |
| Email | 254 символа | Контактный email |
| Телефон | 50 символов | Контактный телефон |
| Object ID | 255 символов | Внешний идентификатор объекта |
| User ID | 255 символов | Внешний идентификатор пользователя |

Запросы, превышающие лимиты, возвращают `400 Bad Request` с кодом ошибки `INVALID_INPUT`.

### Валидация S3-ключей

S3-ключи файловых вложений проверяются для защиты от path traversal атак:

- Блокируются последовательности `..` и URL-кодированные варианты (`%2e%2e`)
- Блокируется инъекция null-байта (`%00`)
- Проверяется принадлежность файла к правильному диалогу

## Санитизация ввода

### HTML-контент

Содержимое сообщений санитизируется на сервере с помощью библиотеки [ammonia](https://docs.rs/ammonia):

**Разрешённые теги:** `p`, `br`, `strong`, `em`, `u`, `s`, `a`, `ul`, `ol`, `li`, `blockquote`, `code`, `pre`, `span`

**Удаляются:** `<script>`, обработчики событий, `javascript:` URL, iframe, формы.

### Клиентская санитизация

Vue SDK использует [DOMPurify](https://github.com/cure53/DOMPurify) для дополнительной клиентской санитизации перед рендерингом HTML. Это обеспечивает защиту в глубину от XSS.

## Безопасность вебхуков

- Заголовок `X-Webhook-Signature` содержит `sha256=<hex-digest>`
- Дайджест вычисляется над телом запроса с использованием `WEBHOOK_SECRET`
- Constant-time сравнение используется внутренне
- Всегда верифицируйте подпись на вашем эндпоинте

## Безопасность файлов

- Типы файлов валидируются по whitelist MIME-типов
- Размер ограничен 100 МБ
- Файлы хранятся в S3 с уникальными UUIDv7-ключами
- URL скачивания presigned и временные
- S3-бакет настроен без публичного доступа

## Чек-лист для продакшена

- [ ] Установить сильный `ADMIN_API_TOKEN` (минимум 32 символа)
- [ ] Включить JWT-аутентификацию (`JWT_AUTH_ENABLED=true`) и установить сильный `JWT_SECRET` (минимум 32 символа)
- [ ] Установить сильный `WEBHOOK_SECRET`
- [ ] Разместить MTChat за reverse proxy с TLS (HTTPS)
- [ ] Ограничить CORS доменами вашего приложения
- [ ] Генерировать JWT-токены на сервере и передавать их в SDK через `token` в конфигурации
- [ ] Установить подходящий `RUST_LOG` для продакшена
- [ ] Настроить `S3_PUBLIC_ENDPOINT` на HTTPS
- [ ] Мониторить `/health` и `/health/ready`
- [ ] Включить встроенный rate limiting (`RATE_LIMIT_ENABLED=true`) и/или rate limiting на уровне reverse proxy

## Будущие улучшения

Запланированные функции безопасности:

- **Фильтрация WebSocket-подписок** (пользователи получают события только для своих диалогов)
- **Rate limiting по пользователю и по эндпоинту** в дополнение к текущему глобальному token-bucket лимитеру
- **CSRF-защита** для не-API эндпоинтов
